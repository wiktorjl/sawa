# Log scan — 2026-05-13

Scanned **162** log files totaling **5.69 GB** in `logs/`. This was a one-shot
audit done while adding NTFY/monitored_run wiring; raw files were deleted
after the scan to reclaim disk. Future log review should use the
`sawa logs` subcommand against `~/.sawa/logs/`.

## Findings (action items)

1. **Postgres connection pool exhaustion in intraday on 2026-04-16.** A single
   intraday run (`intraday_20260416_133002.log`) generated 126 `FATAL: sorry,
   too many clients already` errors. The websocket bar-handler is opening new
   `psycopg.connect()` calls per write without reusing a pool. Reproduce, then
   migrate `sawa/api/websocket_client.py` to the shared connection pool used
   by `mcp_server/database.py` (or a dedicated SyncConnectionPool). Until then,
   a long market session can saturate Postgres and stall every other sawa
   process.

2. **Postgres outage on 2026-04-04 went unalerted.** Six consecutive daily-run
   files between 21:00 and 22:15 ET show `Daily update failed: Connection
   refused`. The Python layer logged but did not notify — the new
   `monitored_run` wrapper now alerts on this, but it is worth verifying the
   scheduler's lock + retry behavior so a flapping DB doesn't suppress
   subsequent alerts.

3. **`^VIX` and `^VIX3M` news errors (RESOLVED in commit `2d4e350`).** 11
   pre-2026-05-13 weekly runs produced `Failed to fetch news for ^VIX:
   Invalid ticker format`. Commit `2d4e350` ("consolidate VIX to single
   source in market_internals") dropped the caret tickers from
   `companies`/`stock_prices`/`technical_indicators` and removed the mirror,
   so `load_news` no longer iterates over them. Listed here only so the
   raw-error counts above stay interpretable.

4. **Logs were unbounded and unrotated.** 5.69 GB in 162 files, with single
   intraday files reaching 112 MB. Default log path is now
   `~/.sawa/logs/` and `setup_logging` uses size-bounded rotation. The MCP
   audit log (`execute_query.log`) was also moved out of the repo.
